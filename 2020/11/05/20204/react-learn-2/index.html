第三章、前端架的分析框

1、前端框架分析

框指的是约束，架指的是支撑。框架会控制我们书写代码各个部分的结构。

2、前端框架解决的问题

代码层面问题

当代码少的时候，感觉不出来；但当代码几千上万行，就会有很多问题；比如结构不清晰，耦合严重。框架通常都是结构化的，无论是MVC\MVVM\模块化开发，都在一定程度上对代码做了分类和规划，降低了混乱程度。

js可以像后端语言那样，分文件写代码，代码长度和工作量都得到了解放。改错的位置也就越来越少。

原生开发里要负责工具、负责架构、负责业务，对开发人员要求较高。

效率问题

多页应用问题

在使用框架前，都是使用浏览器自带的路由，这种路由是在很多页面之间进行跳转，所以叫多页应用；单页应用的HTML页面只有一个，只是在同一个页面里用js来实现路由的功能，达到计划显示内容的效果。

传统的多页路由上，我们每跳转一个新页面，就要先加载HTML，然后再把HTML引入的文件都加载一遍，最后都把他们渲染出来。这个加载过程中占用时间最多的一般就是网络请求，所以多页应用如果重新加载所有东西的时候，加载时间长，导致页面不流畅，造成体验不顺畅的问题。

如果在单页应用里，我们切换页面时，我们需要请求的就是数据，而HTML\JS\CSS这些资源通常都是提前加载好的，页面显示和加载都会很快，省去了很多网络请求，提高用户体验。

多页应用中切换路由时，前一个页面会立即销毁，所以想要动画就无法实现。

多页应用浪费服务器资源。

框架开发的不足

使用框架最大的问题就是会受到实际情况的约束，因为框架里为了实现一些方便的或者是高级的功能，对低版本的浏览器兼容性不是很好。另外单页面应用只有一个页面，内容还都是用js渲染出来的，对目前的搜索引擎是不太友好的

框架的设计是有针对性的，不可能做出来一个全能的框架，每个框架都有自己的定位、自己的适用场景。

框架都是黑盒的，我们再使用的时候都是用他们的一些api或者一些几口，而他内部的实现对我们来说是不可见的，如果框架本身有问题，那对我们就比较麻烦。

框架有学习成本。在使用一个新框架时我们就要了解他的设计思路和api的用法。但磨刀不误砍柴工，对后期很有帮助，同时还能学到框架的设计思想

第四章、开发环境的搭建

1、开发框架搭建

React 框架

官网：https://react.docschina.org/docs/getting-started.html

初识React

他是一个处理用户界面的视图层框架，也就是react本身就是个只负责显示部分的框架，严格说来他都不能算作一个框架，决定了react本身是不太复杂的，不像angular那样要处理各种识事务。

这个框架的组成方式抛弃了传统的继承方式，而是使用了组件化的方式来开发。

比较有特点的地方是他使用了JSX表达式，这是react独有的，我们用它来处理数据和DOM之间的关系，而不用我们再去操作DOM。

虚拟DOM是一种机制，简单的说只有在必要的时候，才回去操作DOM，他通过减少低效的操作，来提高整体的性能

视图层框架

  • 一个构建用户界面的框架

react本身提供的，就是处理页面显示的功能，跟显示有关的主要是两个事，一个是，能把数据变成DOM显示出来，另外就是，处理在DOM上发生的事件，把DOM事件再返回给数据。

Angular或vue这种双向数据绑定框架里，这两件事是放在一起做的，React里这是两个分开的步骤，

  • 声明式的框架

在他的里面你只要声明，你只要声明，你想要干什么，就足够了，剩下的具体怎么做，React框架会帮你完成。 他不用像命令式编程那样，指挥他怎么渲染、怎么调用之类的。

比如在使用JQuery的时候，需要先指挥选择器，选择哪一个元素，然后再操作他，这就是命令式的操作。

React中，我们只要定义好数据和DOM之间的对应关系，数据改变的时候，DOM会自动变化。

  • 数据驱动DOM，再用事件反馈给数据

数据变了，DOM就会跟着变，而DOM上的变化，不像双向绑定那样，直接反馈给数据，需要通过事件，把DOM的变化返回给数据。

组件化开发

  • 组件组合而不是继承

推荐用组件的方式，而不是传统那样，面向对象的方式，通过继承，实现代码的封装，因为组合相对于继承，更灵活。通过组合的方式，也不会受到继承里的约束。

  • state && props

state是组件里的数据，称之为状态。props做父子组件通讯用。子组件拿到父组件的数据。react的组件间，数据传递的方式，是由上向下的，需要父组件，在props里，把操作的方法，传给子组件。

  • 生命周期

一个组件从建立，到存在，最后到销毁的过程，也就是这个组件的生命周期。react提供了灵活的生命周期节点。

理解JSX

一种JS扩展的表达式

HTML模板是先定义好模板，这样只能改变，模板定义好的结构；JSX实际上是个脚本，他随时可以改变，相对于HTML模板更灵活。

虚拟DOM

是框架底层的机制。我们会把定义好的JSX标记，转换成虚拟的DOM，存在内存里，因为内存的读取速度，比操作DOM快的多，

用户做了对DOM可能产生影响的操作，虚拟DOM会对比操作前后的数据。不到万不得已，不做低效的DOM操作。

优势：只有数据变化才会操作DOM；还会整合DOM操作， 这种机制，减少DOM渲染，提高应用性能。

React缺点

编程的思维方式要跟着转过来，比如JSX，单向数据流，之前的框架开发，和原生开发里，都没有接触过。

React本身只是一个视图层的框架，甚至不能成为框架，我们需要依赖他的生态系统，比如处理单页路由，需要React-router，处理数据我们需要用，有了这些才能发挥更强大的能力。也意味着，我们需要了解更多知识。

更新频繁，经常不向前兼容，更新后容易踩坑。
